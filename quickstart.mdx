This guide walks you through creating a Pupload project, defining a simple image processing flow, and running it locally.

## Initialize a new project

<Steps>
  <Step title="Create project directory">
    Create a new directory for your project:

    ```bash
    mkdir my-pupload-project
    cd my-pupload-project
    ```
  </Step>

  <Step title="Initialize the project">
    Initialize a new Pupload project:

    ```bash
    pup init my-project
    ```

    This creates the basic project structure with configuration files and example flows.
  </Step>

  <Step title="Explore the structure">
    Your project now contains:

    ```
    my-pupload-project/
    ├── config/
    │   ├── controller.yaml
    │   └── worker.yaml
    ├── flows/
    │   └── (your flow definitions)
    └── nodedefs/
        └── (your node definitions)
    ```
  </Step>
</Steps>

## Understanding flows and nodes

A **flow** defines your processing pipeline. Here's an example flow that processes images:

```yaml flows/image-resize.yaml
Name: image-resize

Stores:
  - Name: ImageStore
    Type: s3local
    Params:
      BucketName: images

DefaultStore: ImageStore

AvailableNodes:
  - github.com/pupload/pupload/ffmpeg

DataWells:
  - Store: ImageStore
    Edge: image-in
    Type: dynamic

Nodes:
  - ID: 0
    DefName: github.com/pupload/pupload/ffmpeg
    Inputs:
      - Name: VideoStream
        Edge: image-in
    Outputs:
      - Name: VideoOut
        Edge: image-out
    Flags:
      - Name: Flag
        Value: resize
    Command: Encode
```

<Info>
**DataWells** define entry points where files enter your pipeline. The `Edge` connects to node inputs.
</Info>

A **node definition** specifies what a processing step can do:

```yaml nodedefs/ffmpeg.yaml
ID: 0
Publisher: pupload
Name: ffmpeg
Image: linuxserver/ffmpeg

Inputs:
  - Name: VideoStream
    Description: Input video/image stream
    Required: true
    Type:
      - video
      - photo

Outputs:
  - Name: VideoOut
    Description: Processed output
    Required: true
    Type:
      - video
      - photo

Flags:
  - Name: Flag
    Description: Processing flag
    Required: true
    Type: false

Command:
  Name: Encode
  Description: Encode media
  Exec: -f image2 -i ${VideoStream} -vcodec mjpeg -f image2 ${VideoOut}
```

## Run your first flow

<Steps>
  <Step title="Start development mode">
    The easiest way to test a flow is using the development server with mock storage:

    ```bash
    pup test image-resize --input image-in=./path/to/image.jpg --mocks3
    ```

    <Note>
    The `--mocks3` flag creates a mock S3 server automatically. No real storage configuration needed!
    </Note>

    This command:
    - Starts a local controller and worker
    - Creates mock S3 storage
    - Uploads your input file
    - Runs the flow
    - Shows progress in real-time
  </Step>

  <Step title="Watch the output">
    You'll see output like:

    ```bash
    Starting local development environment...
    Controller running on http://localhost:1234
    Worker connected
    Flow started: image-resize
    Node 0: Processing...
    Node 0: Complete
    Flow completed successfully
    ```
  </Step>

  <Step title="Enable TUI mode (optional)">
    For a better visual experience, use the terminal UI:

    ```bash
    pup test image-resize --input image-in=./image.jpg --mocks3 --tui
    ```

    This displays an interactive interface showing flow progress, node status, and logs.
  </Step>
</Steps>

## Run with persistent dev server

For iterative development, run the dev server separately:

<Steps>
  <Step title="Start the dev server">
    In one terminal, start the development server:

    ```bash
    pup dev
    ```

    This starts both the controller (API server) and a worker process.
  </Step>

  <Step title="Test via CLI">
    In another terminal, run flows against the dev server:

    ```bash
    pup test image-resize --input image-in=./image.jpg --remote localhost:1234
    ```

    The `--remote` flag connects to your running dev server instead of starting a new one.
  </Step>

  <Step title="Test via API">
    You can also trigger flows using the REST API:

    ```bash
    curl -X POST http://localhost:1234/v1/projects/<project_id>/flows/image-resize
    ```
  </Step>
</Steps>

<Warning>
When using `--remote` or the API, make sure to handle file uploads separately. The dev server expects files to be available in the configured storage.
</Warning>

## Managing controllers

For production deployments, add remote controllers:

```bash Controller commands
# Add a controller
pup controller add production https://controller.example.com

# List all controllers
pup controller list

# Remove a controller
pup controller remove production
```

<Info>
Controllers are saved to your project configuration and can be referenced by name when running flows.
</Info>

## Configuration files

Customize your deployment with configuration files:

<Tabs>
  <Tab title="Controller">
    Edit `config/controller.yaml`:

    ```yaml config/controller.yaml
    controller:
      port: 8080
      redis:
        addr: localhost:6379
      storage:
        type: s3
        bucket: uploads
        endpoint: s3.amazonaws.com
    ```
  </Tab>

  <Tab title="Worker">
    Edit `config/worker.yaml`:

    ```yaml config/worker.yaml
    worker:
      redis:
        addr: localhost:6379
      docker:
        enabled: true
      resources:
        - c-medium    # CPU tasks
        - g-small     # GPU tasks
    ```

    <Tip>
    Workers only pick up tasks matching their subscribed resource tiers. Scale GPU workers separately from CPU workers!
    </Tip>
  </Tab>
</Tabs>
