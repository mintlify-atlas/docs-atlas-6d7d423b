This example demonstrates a complete ML inference pipeline that handles image uploads, preprocessing, GPU inference, and postprocessing for object detection or image classification tasks.

## Flow overview

The pipeline processes images through these stages:

<Steps>
  <Step title="Upload">
    User uploads an image for inference
  </Step>
  <Step title="Preprocess">
    Resize, normalize, and prepare image for model input
  </Step>
  <Step title="GPU inference">
    Run the ML model on GPU-enabled workers
  </Step>
  <Step title="Postprocess">
    Parse model output, apply NMS, format results
  </Step>
  <Step title="Visualize">
    Draw bounding boxes and labels on the original image
  </Step>
  <Step title="Store results">
    Save predictions and annotated image
  </Step>
</Steps>

    Source: static

  - Store: InferenceResults

  - Store: InferenceResults

      - Name: ImageIn
        Edge: image-input
    Outputs:
      - Name: ProcessedImage
        Edge: preprocessed-image
    Command: Preprocess
    Flags:
      - Name: TargetSize

      - Name: Normalize

  # Run GPU inference
  - ID: inference-engine

      - Name: ImageTensor
        Edge: preprocessed-image
      - Name: ModelWeights
        Edge: model-weights
    Outputs:
      - Name: RawPredictions
        Edge: raw-predictions
    Command: Infer
    Flags:
      - Name: Framework
        Value: "pytorch"
      - Name: Device
        Value: "cuda"
      - Name: BatchSize
        Value: "1"
      - Name: Precision

  # Postprocess predictions
  - ID: postprocessor
    Uses: github.com/pupload/pupload/ml-postprocess
    Inputs:
      - Name: RawPredictions
        Edge: raw-predictions
      - Name: OriginalImage
        Edge: image-input
    Outputs:
      - Name: Predictions
        Edge: predictions-json
    Command: Postprocess
    Flags:
      - Name: Task
        Value: "object-detection"
      - Name: ConfidenceThreshold
        Value: "0.5"

  - ID: visualizer
    Uses: github.com/pupload/pupload/ml-visualize
    Inputs:
      - Name: ImageIn
        Edge: image-input
      - Name: Predictions
        Edge: predictions-json

      - Name: AnnotatedImage

    Command: DrawBoxes
    Flags:

## Node details

### Preprocessor

Prepares the input image for the ML model:
- Resizes to model's expected input size (e.g., 640x640 for YOLOv8)
- Normalizes pixel values using ImageNet statistics
- Converts to tensor format (CHW layout)
- Optionally applies augmentations for test-time augmentation

- **Resource tier**: `c-small` (CPU preprocessing)
- **Output**: Preprocessed image tensor

<Tip>
**Batch processing**: For high throughput, modify the preprocessor to output batches of multiple images and adjust the inference batch size accordingly.
</Tip>

### Inference engine

Runs the ML model on GPU workers. This is the most computationally expensive step.

- **Resource tier**: `g-small`, `g-medium`, or `g-large` (GPU required)
- **Supported frameworks**: PyTorch, TensorFlow, ONNX Runtime
- **Precision options**:
  - `fp32`: Full precision (highest accuracy)
  - `fp16`: Half precision (2x faster, minimal accuracy loss)
  - `int8`: Quantized (4x faster, slight accuracy loss)

**GPU requirements**:
```yaml worker.yaml
worker:
  resources:
    - g-small  # NVIDIA GPU workers
  docker:
    runtime: nvidia
    limits:
      gpus: "1"
      memory: "8GB"
```

<Note>
**Model loading**: The model weights are loaded from a static datawell. Upload your trained model to the ModelStore S3 bucket before running inference.
</Note>

### Postprocessor

Parses raw model outputs and applies postprocessing:

- **Non-maximum suppression (NMS)**: Removes duplicate detections
- **Confidence filtering**: Keeps only high-confidence predictions
- **Coordinate transformation**: Scales boxes back to original image dimensions
- **Class mapping**: Maps class IDs to human-readable labels

- **Resource tier**: `c-small` (CPU postprocessing)
- **Output**: JSON with structured predictions

**Example output**:
```json
{
  "predictions": [
    {
      "class": "person",
      "confidence": 0.94,
      "bbox": [120, 50, 250, 400],
      "class_id": 0
    },
    {
      "class": "dog",
      "confidence": 0.87,
      "bbox": [300, 200, 450, 380],
      "class_id": 16
    }
  ],
  "inference_time_ms": 23,
  "num_detections": 2
}
```

### Visualizer

Draws bounding boxes, labels, and confidence scores on the original image for visual verification.

- **Resource tier**: `c-small`
- **Output**: Annotated JPEG image
- **Features**: Auto-colored boxes per class, confidence scores, customizable styling

## Performance optimization

### GPU utilization

**Key factors for GPU efficiency**:

1. **Batch size**: Increase batch size to fully utilize GPU memory
   - Single image: ~20% GPU utilization
   - Batch of 8: ~80% GPU utilization
   - Batch of 16: ~95% GPU utilization

2. **Model precision**: Use FP16 for 2x speedup with minimal accuracy loss

3. **Model optimization**: Use TensorRT or ONNX Runtime for optimized inference

### Cost optimization

GPU workers are expensive. Optimize costs:

```yaml
# CPU preprocessing (cheap)
preprocessor:
  resource_tier: c-small

# GPU inference (expensive, but necessary)
inference-engine:
  resource_tier: g-small

# CPU postprocessing (cheap)
postprocessor:
  resource_tier: c-small
```

Only the inference node requires GPU. Pre/postprocessing runs on cheap CPU workers.

### Throughput optimization

For high-throughput scenarios:

1. **Queue batching**: Modify the preprocessor to accumulate images into batches
2. **Dynamic batching**: Let the inference engine wait up to 100ms to fill a batch
3. **Multiple GPU workers**: Scale GPU workers horizontally

```yaml worker.yaml
worker:
  resources:
    - g-small
  scaling:
    min_workers: 2
    max_workers: 10
    target_queue_depth: 5
```

## Model examples

### Object detection (YOLOv8)

The example above uses YOLOv8 for object detection. Detects multiple objects with bounding boxes.

**Use cases**: Security cameras, autonomous vehicles, retail analytics

### Image classification (ResNet)

For classification tasks, simplify the flow:
- Remove bounding box visualization
- Postprocessor outputs top-k class probabilities
- No NMS required

**Use cases**: Content moderation, medical imaging, product categorization

### Semantic segmentation

For pixel-level segmentation:
- Postprocessor generates segmentation masks
- Visualizer overlays colored masks
- Larger model outputs (H×W masks)

**Use cases**: Medical imaging, autonomous driving, satellite imagery

### Image generation (Stable Diffusion)

For generative models:
- Input is text prompt instead of image
- Preprocessing converts text to embeddings
- Output is generated image
- No postprocessing needed

**Use cases**: Creative tools, product design, synthetic data generation
