## Overview

Pupload uses a distributed controller/worker architecture to orchestrate data processing flows. This design separates flow orchestration from execution, enabling horizontal scaling and efficient resource utilization.

## Architecture components

## Controller

The controller is the brain of Pupload. It manages flow definitions, schedules nodes to workers, and tracks execution state.

### Controller responsibilities

- **Flow management**: Load and validate flow definitions
- **State tracking**: Monitor flow and node execution states
- **Node scheduling**: Assign ready nodes to available workers
- **API service**: Provide HTTP endpoints for flow submission and monitoring
- **Storage coordination**: Manage datawell URLs and artifact tracking

### Controller structure

```go
func RunWithConfig(ctx context.Context, cfg *config.ControllerConfig) error {
    // Initialize logging and telemetry
    logging.Init(logging.Config{
        AppName: "pupload.controller",
        Level:   slog.LevelInfo,
        Format:  "text",
    })

    // Create SyncPlane for controller/worker coordination
    s, err := syncplane.CreateControllerSyncLayer(cfg.SyncPlane)
    if err != nil {
        return err
    }
    defer s.Close()

    // Create flow service
    f, err := flow.CreateFlowService(cfg, s)
    if err != nil {
        return err
    }
    defer f.Close(ctx)

    // Create project service
    p, err := projects.CreateProjectService(cfg.ProjectRepo)
    if err != nil {
        return err
    }
    defer p.Close()

    // Start HTTP server
    handler := controllerserver.NewServer(*cfg, f, p)
    srv := &http.Server{
        Addr:    ":1234",
        Handler: handler,
    }

    go func() {
        if err := srv.ListenAndServe(); err != http.ErrServerClosed {
            log.Error("server forced to shutdown", "err", err)
        }
    }()

    <-ctx.Done()
    return srv.Shutdown(shutdownCtx)
}
```

## Worker

Workers are the muscle of Pupload. They execute nodes by running containers, transferring data, and reporting results back to the controller.

### Worker responsibilities

- **Resource management**: Track available CPU, memory, storage, and GPU
- **Node execution**: Run containerized workloads with proper isolation
- **Data transfer**: Download inputs and upload outputs using presigned URLs
- **Health reporting**: Advertise available resource tiers to the controller

### Worker structure

```go
func RunWithConfig(ctx context.Context, cfg *config.WorkerConfig) error {
    // Initialize logging
    logging.Init(logging.Config{
        AppName: "worker",
        Level:   slog.LevelInfo,
        Format:  "text",
    })

    log := logging.Root()

    // Initialize telemetry
    telemetry.Init(cfg.Telemetry, "pupload.worker")

    // Create SyncPlane for worker communication
    s, err := syncplane.CreateWorkerSyncLayer(cfg.SyncPlane, cfg.Resources)
    if err != nil {
        return err
    }

    log.Info("Worker starting up...")
    
    // Create container service
    cs := container.CreateContainerService()

    // Create resource manager
    rm, err := resources.CreateResourceManager(cfg.Resources)
    if err != nil {
        return err
    }

    // Start worker server
    server.NewWorkerServer(s, &cs, rm)

    <-ctx.Done()

    log.Info("Worker shutting down...")
    return s.Close()
}
```

## SyncPlane

The SyncPlane is the communication layer between controllers and workers. It handles job queuing, state synchronization, and worker discovery.

### Current implementation: Redis

## Flow execution flow

Here's how a flow moves through the system:

1. **Submission**: User submits a flow via the controller's HTTP API
2. **Initialization**: Controller creates a `FlowRun` and initializes datawells
3. **Waiting**: Flow waits for input data to be uploaded (if needed)
4. **Scheduling**: Controller identifies ready nodes and publishes them to the SyncPlane
5. **Claiming**: Workers with appropriate resources claim nodes from the queue
6. **Execution**: Worker runs the node in a container with proper resource limits
7. **Completion**: Worker uploads outputs and reports success/failure to controller
8. **Next nodes**: Controller marks downstream nodes as ready and schedules them
9. **Completion**: When all nodes finish, flow is marked as complete

### State machine

Flows progress through these states:

```go
switch rt.FlowRun.Status {
case models.FLOWRUN_STOPPED:
    // Initial state

case models.FLOWRUN_WAITING:
    rt.updateWaiting()
    rt.updateAllNodes()

    if len(rt.nodesReady()) == 0 {
        return
    }

    rt.FlowRun.Status = models.FLOWRUN_RUNNING

case models.FLOWRUN_RUNNING:
    for _, nodeID := range rt.nodesReady() {
        if err := rt.handleExecuteNode(ctx, nodeID, s); err != nil {
            rt.FlowRun.Status = models.FLOWRUN_ERROR
            return
        }

        rt.FlowRun.NodeState[nodeID] = models.NodeState{
            Status: models.NODERUN_RUNNING,
            Logs:   rt.FlowRun.NodeState[nodeID].Logs,
        }
    }

    rt.FlowRun.Status = models.FLOWRUN_WAITING

case models.FLOWRUN_COMPLETE:
    return

case models.FLOWRUN_ERROR:
    return
}
```

## Scaling patterns

### Horizontal controller scaling

Run multiple controller instances behind a load balancer:

```bash
# Controller 1
pupload controller --config /etc/pupload/controller.yaml

# Controller 2  
pupload controller --config /etc/pupload/controller.yaml

# Controller 3
pupload controller --config /etc/pupload/controller.yaml
```

All controllers share state via Redis, ensuring consistent behavior.

### Heterogeneous worker pools

Deploy workers with different resource profiles:

```yaml
# CPU-optimized worker
Resources:
  MaxCPU: "32"
  MaxMemory: "64gb"
  MaxStorage: "500gb"
```

```yaml
# GPU worker
Resources:
  MaxCPU: "16"
  MaxMemory: "96gb"
  MaxStorage: "1tb"
Runtime:
  EnableGPUSupport: true
```

```yaml
# Memory-optimized worker
Resources:
  MaxCPU: "8"
  MaxMemory: "128gb"
  MaxStorage: "250gb"
```

Workers automatically advertise their capabilities, and the controller schedules nodes appropriately.

<Warning>
Ensure your Redis instance is properly sized for your workload. High-throughput deployments may need Redis clustering or a managed Redis service.
</Warning>
